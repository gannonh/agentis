Support-agent API flow
01 / Request
Client sends chat request
SupportAgentChatRequest carries agent, conversation, message, question, and selected knowledge source references.
02 / Route
Worker or Vite route receives it
Both paths use /api/support-agent/respond; Cloudflare also exposes /health.
03 / Handler
Agentis validates shape
createSupportAgentApiHandler rejects wrong methods and invalid request bodies.
04 / Runtime
Server-side runtime runs
Runtime resolves provider config, builds prompt with documentation context, and calls the model gateway.
05 / Response
Public answer returns
SupportAgentChatResponse includes answer, sources, and public runtime metadata. Secrets stay server-side.
Runtime and ownership boundaries
Browser-visible

Configuration and selected knowledge

The browser can prepare a hosted config handoff with template identity, selected knowledge source IDs, context references, runtime adapter name, and deployment intent.

No API keysNo raw deploy secret
Server-side

Provider credentials and model calls

The local dev route reads `.env`; the Cloudflare Worker reads Worker secrets. Both map credentials into the same support-agent handler shape.

OpenAI key in secret bindingpublic runtime metadata only
Agentis-owned

Contracts, adapter boundary, docs context

Agentis owns request/response contracts, failure mapping, source provenance, deploy command scripts, and the policy that secrets never enter browser state.

Setup path
  • Authenticate Wrangler with Cloudflare account or API token.
  • Register gannonh-agentis.workers.dev workers.dev subdomain.
  • Deploy smoke Worker to prove Cloudflare publish path.
  • Upload SUPPORT_AGENT_OPENAI_API_KEY and SUPPORT_AGENT_DEPLOYMENT_SECRET as Worker secrets.
  • Deploy apps/web/src/worker/support-agent-worker.ts through scripts/deploy-cloudflare-support-agent-worker.sh.
